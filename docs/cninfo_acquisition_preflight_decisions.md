# Authorized CNINFO Acquisition — Architecture Closure Decisions

## Authority

This document is an authoritative amendment to Issue #188 and `docs/cninfo_acquisition_preflight.md`.

Where this document is more specific than the main preflight, this document controls the future implementation contract. It exists to close two infrastructure and chronology choices found during the first fixed-head architecture review.

No production implementation, migration, dependency, credential or live request is authorized here.

## Decision 1 — Raw objects use database binary storage in v1

The v1 implementation must persist immutable raw metadata response bytes and announcement document bytes directly in the reviewed PostgreSQL/SQLite database schema.

- SQLAlchemy representation: `LargeBinary`.
- PostgreSQL representation: `BYTEA`.
- SQLite representation: `BLOB`.
- `disclosure_raw_metadata_objects` owns the exact metadata response bytes.
- `disclosure_raw_document_objects` owns the exact official document bytes.
- No filesystem path, object-store URI, external blob service or hybrid storage mode is permitted in v1.
- No user-supplied storage locator is accepted.

### Size ceilings

- Maximum raw metadata response body: 5 MiB per object.
- Maximum raw announcement document: 25 MiB per object.
- Maximum aggregate raw bytes committed by one acquisition command: 250 MiB.

Objects above a ceiling fail closed with `raw_object_too_large`. The application may persist credential-safe attempt diagnostics, but it must not truncate, partially store or substitute the object.

### Atomicity and lifecycle

- Each raw object row stores bytes, SHA-256, byte length and provenance in the same database transaction.
- A row cannot reference bytes that are absent from the same database.
- Backup, restore and migration behavior follows the database backup boundary.
- Populated downgrade refusal covers the binary rows before any table drop.
- A future object-store migration requires a separate Strict Architecture Preflight and cannot be introduced as an implementation refactor.

## Decision 2 — Live write timestamps are system-owned

A live acquisition write must not accept a user-selected `recorded_at_utc`, `requested_at_utc` or `fetched_at_utc`.

The future production command accepts:

- exact source authorization revision;
- explicit selector/date/request ceilings;
- information cutoff;
- remote-access confirmation;
- credential profile key;
- optional expected-latest attempt revision;
- dry-run flag.

Time ownership is:

- `requested_at_utc`: generated by the application immediately before the first transport operation;
- `fetched_at_utc`: generated by the application when each response/document body has been received and validated for the capture boundary;
- `recorded_at_utc`: generated by the persistence boundary when the revision/object is written;
- `reviewed_at_utc`: generated by the review-command persistence boundary;
- Evidence Ledger accepted-record time: generated by the atomic acceptance transaction.

### Clock rules

- All generated timestamps are timezone-aware UTC.
- `requested_at_utc <= fetched_at_utc <= recorded_at_utc` for each captured object.
- Review time cannot precede the referenced recorded time.
- Evidence acceptance time cannot precede the review time.
- A generated time cannot be supplied or overridden by ordinary production JSON.
- Tests may inject one explicit deterministic clock through a test-only dependency boundary; production code cannot expose that override through CLI or API fields.
- A source publication time remains source-owned and may precede acquisition.
- Information cutoff remains an explicit user/research selector and cannot exceed the source publication boundary accepted by the command.

### Dry-run behavior

A dry run performs no network or persistence write and therefore returns no production `requested_at_utc`, `fetched_at_utc` or `recorded_at_utc`. It returns only the canonical redacted request plan and request fingerprint.

## Decision 3 — Contract evidence precedes the implementation Issue

The future production implementation Issue must not be opened merely because PR #189 is approved or merged.

Before an implementation Issue is created, the project owner must record or confirm the complete non-secret source-contract facts required by Issue #188, including:

1. official service/API documentation label and effective version/date;
2. automated-access entitlement;
3. retention and local binary-storage rights;
4. credential mechanism and approved host set;
5. endpoint/method/request/response schema contracts;
6. pagination, ordering, quotas and rate limits;
7. stable announcement and document-object identifiers;
8. correction/supersession behavior;
9. contract expiry and suspension rules.

Confidential documents and credentials remain outside the repository. The repository may record a non-secret contract label, review date, approved capability list and contract-package fingerprint where permitted.

If those facts are unavailable, the architecture may be merged as a governed source decision, but production implementation remains explicitly blocked.

## Updated golden-path requirement

The implementation golden path must prove database-owned binary capture and system-owned chronology:

1. fixture transport returns one metadata body below 5 MiB and one document below 25 MiB;
2. exact bytes, SHA-256 and byte length commit into database binary columns;
3. generated UTC times satisfy request/fetch/record chronology;
4. identical replay is idempotent;
5. oversized metadata/document objects fail closed without truncation;
6. human review and Evidence Ledger acceptance occur only after capture and use system-owned times;
7. rollback leaves no partial EvidenceItem, claim link or provenance bridge;
8. ordinary tests and fixture demo use the injected test clock and zero live network.

## Updated primary failure requirements

The future implementation must fail closed for:

- `raw_metadata_too_large`;
- `raw_document_too_large`;
- `aggregate_capture_too_large`;
- `raw_length_mismatch`;
- `raw_fingerprint_mismatch`;
- `generated_chronology_invalid`;
- any production input containing a forbidden generated-time override;
- any attempt to use filesystem/object-store storage in the v1 command path.
